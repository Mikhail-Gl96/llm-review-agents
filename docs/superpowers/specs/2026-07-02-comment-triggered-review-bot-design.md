# Спецификация: бот-диспетчер ревью по комментарию (SP2)

Дата: 2026-07-02
Статус: на согласовании
Связано с: [SP1 — тулкит ревьюеров](2026-07-02-reviewers-toolkit-design.md)

## 1. Контекст и цель

SP1 даёт два ревьюера в двух режимах: **локально** и **CI-автомат** (на каждый PR).
SP2 добавляет **третий режим — «по требованию / интерактивный»**: ревью запускается
**только когда человек позвал бота в комментарии** к PR/MR, не раньше.

Реализуется как **центральный бот-сервис** (webhook): деплоится один раз, обслуживает
много репозиториев, маршрутизирует запрос к нужному ревьюеру по имени в комментарии.
Модуль в этой же репе — `bot/`.

## 2. Требования

1. **Триггер по комментарию.** Ревью стартует только при упоминании бота в комменте PR/MR.
2. **Маршрутизация по имени** (реестр `имя → движок`):
   - `@review.pr.agent` → pr-agent
   - `@review.ocr` → ocr
   - `@review` (без суффикса) → **дефолт = ocr**
3. **Расширяемость (первого класса):** добавить ревьюера = один адаптер + строка в
   реестре + имя; добавить платформу = один платформенный адаптер. Диспетчер при этом
   не трогается.
4. **Платформы v1:** GitHub + GitLab. Bitbucket — позже через тот же интерфейс.
5. **Централизованность:** один деплой обслуживает много репозиториев.

## 3. Не входит в объём (YAGNI для v1)

- **Bitbucket** (интерфейс адаптера закладываем, реализацию — потом).
- **Serverless-режим** (`issue_comment`-workflow без сервера) — отдельная опция, не сейчас.
- **Интерактивные под-команды** сверх выбора движка (`describe/improve/ask`, флаги
  вроде `--commit`) — задел на будущее, в v1 только «ревью текущего PR».
- **UI/дашборд, БД, аналитика.**

## 4. Терминологический нюанс: как работает «тег»

`@review.ocr` — **не** отдельный аккаунт (в юзернеймах GitHub нет точек). Это **один
бот-app на платформу**, который получает события по всем комментам PR и **парсит текст**
на токен `@review[.<движок>]`. Маршрут — по суффиксу; `@review` без суффикса → дефолт.
На GitLab та же схема (Note-хук + парсинг тела). То есть «тег» — текстовая конвенция,
а не платформенный @mention реального аккаунта.

## 5. Архитектура

Конвейер на каждое событие «новый комментарий»:

```
webhook (приём + verify подписи)
  → игнор комментов самого бота (анти-loop) + дедуп по id события
  → парсинг @review[.движок]  → РЕЕСТР { "ocr": …, "pr.agent": … }  (нет → дефолт ocr)
  → авторизация: кто позвал (allowlist: коллабораторы / участники орг)
  → checkout PR (клон + fetch base и head во временную папку)
  → РЕВЬЮЕР-АДАПТЕР запускает движок в ЛОКАЛЬНОМ режиме (том же, что в SP1)
  → результат ревью в markdown
  → постинг ответом-комментом в тот же PR/MR
  → уборка временной папки
```

Две точки расширения — два реестра адаптеров:

- **Реестр ревьюеров** `{ ключ → reviewer-адаптер }`.
- **Реестр платформ** `{ github, gitlab }` — платформенные адаптеры.

## 6. Компонент: приём вебхуков

- **GitHub:** событие comment на PR (issue-comment на pull request); проверка подписи
  `X-Hub-Signature-256`.
- **GitLab:** Note Hook на merge request; проверка `X-Gitlab-Token`.
- **Общее:** дедуп по id события/комментария (повторные доставки), **игнор комментов,
  автором которых является сам бот** (защита от петли), быстрый `200 OK` на нерелевантные
  события.

## 7. Компонент: парсер/роутер

Регексом достаёт `@review` и опциональный суффикс `.<движок>`; суффикс ищется в реестре
ревьюеров; нет суффикса или неизвестный ключ → дефолтный движок (ocr, настраивается).
Ключи реестра v1: `ocr`, `pr.agent`.

## 8. Компонент: авторизация

Дефолтный **allowlist**: бот отвечает только на запросы от пользователей с правом на
репозиторий (коллабораторы / участники организации), проверка через API платформы.
Настраивается (список логинов / уровень доступа). Причина — иначе любой прохожий жжёт
токены модели.

## 9. Компонент: reviewer-адаптеры

Единый интерфейс:

```
run(checkout_dir, base_ref, head_ref) -> markdown
```

- **ocr-адаптер:** `ocr review --from <base> --to <head>` в чекауте → markdown из stdout.
- **pr-agent-адаптер:** запуск pr-agent в local-режиме против чекаута → содержимое
  `review.md`.
- Оба движка уже умеют диффить локальный git-чекаут (переиспользуем локальный режим SP1).
- **Единый постинг:** адаптер только возвращает markdown; постит сам диспетчер (одинаково
  для всех движков и платформ).
- **Добавить ревьюера** = новый адаптер + строка в реестре + имя. Диспетчер не меняется.

## 10. Компонент: платформенные адаптеры

Единый интерфейс: `verify(request)`, `get_pr_context(event)` (repo, base_ref, head_ref,
pr_id, автор), `checkout(context, token)`, `post_comment(context, markdown, token)`.
v1: `github`, `gitlab`. Bitbucket — реализация позже по тому же интерфейсу.

## 11. Стек и деплой

- **Python + FastAPI** — сервис.
- **Docker-образ с обоими рантаймами:** Python (`pr-agent`) + Node (`ocr`), плюс `git`.
- Переменные окружения: токены платформ (GitHub App/PAT, GitLab token), ключи
  LLM-провайдеров (см. `docs/providers.md` из SP1), дефолтный движок, список включённых
  движков, allowlist, секреты вебхуков.
- v1 обрабатывает событие синхронно, с таймаутом; очередь/воркеры — задел на будущее.

## 12. Безопасность

Verify подписи вебхуков; allowlist на инициатора; анти-loop (игнор своих комментов);
таймаут и лимит размера диффа/ответа; клон приватных репозиториев по токену; никаких
секретов в логах.

## 13. Расширяемость (закрепляет требование №3)

- **Новый ревьюер:** написать адаптер (`run(...) → markdown`), зарегистрировать под
  именем, при желании назначить дефолтом. Документируется в `bot/README.md` и
  перекликается с `docs/add-a-reviewer.md` из SP1.
- **Новая платформа:** реализовать платформенный адаптер под общий интерфейс.

## 14. Открытые технические вопросы (проверить при написании)

- GitHub: App (webhook) vs токен-PAT; точные права (чтение репо, запись коммента);
  формат события issue-comment на PR.
- GitLab: права токена бота на коммент к MR; поля Note-хука.
- Точный способ получить markdown из каждого движка (stdout vs файл `review.md`).
- Лимит длины комментария на платформах (резать/сворачивать длинные ревью).
- Конкурентность: блокировка/таймаут на v1, нужна ли очередь.

## 15. Этапы реализации (v1)

1. Каркас FastAPI + конфиг + реестры (пустые адаптеры).
2. GitHub платформенный адаптер: verify → context → post_comment.
3. Checkout PR (клон base+head во временную папку).
4. ocr-адаптер (дефолт) → сквозной путь «коммент → ревью → ответ» на GitHub.
5. pr-agent-адаптер.
6. Роутер `@review[.движок]` + allowlist.
7. GitLab платформенный адаптер.
8. Docker-образ (python+node+git) + `bot/README.md` (деплой, регистрация вебхука, конфиг).

## 16. Связь с SP1 и порядок

Бот переиспользует локальный режим ревьюеров из SP1, поэтому естественный порядок —
**сначала SP1 (фундамент: конфиги провайдеров, локальный запуск), затем SP2**. Разводящая
страница (корневой README из SP1) должна включать третий режим — ссылку на `bot/`.
